The mirror workflow requires a GitHub personal access token to authenticate and push changes to the destination repository.

## FRITES_2026_MIRROR_TOKEN

This is the primary authentication secret used by the workflow to push mirrored content to GitHub.

<ParamField path="FRITES_2026_MIRROR_TOKEN" type="string" required>
  A GitHub personal access token (classic or fine-grained) with write access to the destination repository.
</ParamField>

### Required permissions

The token must have the following permissions:

<ParamField path="repo" type="scope" required>
  Full control of private repositories. This scope grants:
  - `repo:status` - Access commit status
  - `repo_deployment` - Access deployment status
  - `public_repo` - Access public repositories
  - `repo:invite` - Access repository invitations
  - `security_events` - Read and write security events
</ParamField>

For fine-grained tokens, you need:
- **Repository access**: Only select repositories (Thibault-Monnier/frites-2026)
- **Permissions**: Contents (Read and write)

<Warning>
  The `repo` scope grants broad access to all repositories the token owner can access. Use fine-grained tokens when possible to limit the blast radius if the token is compromised.
</Warning>

### How it's used in the workflow

The token is injected into the git push URL using GitHub Actions secrets syntax:

```yaml
run: |
  cd repo.git
  git push --mirror https://x-access-token:${{ secrets.FRITES_2026_MIRROR_TOKEN }}@github.com/Thibault-Monnier/frites-2026.git
```

The format `https://x-access-token:TOKEN@github.com/...` is GitHub's standard authentication pattern for HTTPS git operations. The username is always `x-access-token`, and the token is provided as the password.

<ParamField path="secrets.FRITES_2026_MIRROR_TOKEN" type="expression">
  GitHub Actions secrets are accessed using the expression syntax `${{ secrets.SECRET_NAME }}`. The secret value is automatically masked in logs.
</ParamField>

## Security best practices

Follow these guidelines to keep your mirror workflow secure:

### Use fine-grained tokens

<Warning>
  Classic tokens with `repo` scope have access to all repositories. Fine-grained tokens can be restricted to a single repository.
</Warning>

When creating a fine-grained token:
1. Set expiration to 90 days or less
2. Limit repository access to only `Thibault-Monnier/frites-2026`
3. Grant only Contents (Read and write) permission

### Store tokens in GitHub Secrets

Never hardcode tokens in workflow files or commit them to version control. Always use GitHub repository secrets:

1. Navigate to your repository Settings
2. Go to Secrets and variables → Actions
3. Click "New repository secret"
4. Name: `FRITES_2026_MIRROR_TOKEN`
5. Value: Your personal access token

### Monitor token usage

Regularly review token usage in your GitHub settings:

- Check the "Last used" timestamp
- Review access logs for unusual activity
- Revoke tokens that haven't been used recently

<Warning>
  If you suspect a token has been compromised, revoke it immediately and create a new one. Update the GitHub secret with the new token value.
</Warning>

## How to rotate tokens

Token rotation is a critical security practice. Follow these steps to rotate the mirror token:

### Step 1: Generate a new token

1. Go to GitHub Settings → Developer settings → Personal access tokens
2. Click "Generate new token" (classic or fine-grained)
3. Name it descriptively (e.g., "Frites 2026 Mirror - March 2026")
4. Select required permissions:
   - Classic: `repo` scope
   - Fine-grained: Contents (Read and write) for the specific repository
5. Set expiration date (recommended: 90 days)
6. Click "Generate token" and copy the value immediately

### Step 2: Update the GitHub secret

1. Navigate to your workflow repository (where the `.github/workflows/gitlab-mirror.yaml` file lives)
2. Go to Settings → Secrets and variables → Actions
3. Find `FRITES_2026_MIRROR_TOKEN` in the list
4. Click the edit icon
5. Paste the new token value
6. Click "Update secret"

<Note>
  The change takes effect immediately. The next workflow run will use the new token.
</Note>

### Step 3: Verify the new token

1. Go to Actions tab in your repository
2. Manually trigger the workflow using "Run workflow"
3. Verify the workflow completes successfully
4. Check that the mirror repository received the latest changes

### Step 4: Revoke the old token

1. Return to GitHub Settings → Developer settings → Personal access tokens
2. Find the old token in the list
3. Click "Revoke" to invalidate it

<Warning>
  Only revoke the old token after confirming the new token works. Revoking prematurely will cause the workflow to fail.
</Warning>

## Troubleshooting

If you encounter authentication errors:

<AccordionGroup>
  <Accordion title="Error: Invalid username or password">
    The token is invalid or has been revoked. Generate a new token and update the secret.
  </Accordion>
  
  <Accordion title="Error: Resource not accessible by personal access token">
    The token lacks required permissions. Ensure it has the `repo` scope (classic) or Contents write permission (fine-grained).
  </Accordion>
  
  <Accordion title="Error: Repository not found">
    The token doesn't have access to the destination repository. Verify the token owner has write access to `Thibault-Monnier/frites-2026`.
  </Accordion>
</AccordionGroup>
